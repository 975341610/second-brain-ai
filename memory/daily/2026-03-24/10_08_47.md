我翻了 chromadb 的源码，在 `chromadb/api/__init__.py` 第 33 行找到了那个"定时炸弹"：

```python
import chromadb.utils.embedding_functions as ef  # ← 就是这行！
```

### 💡 真正的根因

这行代码要求在执行时，`chromadb.utils` 和 `chromadb.utils.embedding_functions` 必须已经在 `sys.modules` 里注册好了。但在 PyInstaller 打包环境下，Python 会去找 `chromadb` 这个父包来解析它——而这时候 `chromadb/__init__.py` 自己还没跑完（正在初始化中），于是就引发了 **"partially initialized module"** 的循环导入死锁。

之前几次修复的失误：我只注入了 `chromadb.utils.embedding_functions`，却**漏了父包 `chromadb.utils`**，导致 Python 还是要走父包路径，循环没有被真正打断。

### ✅ 这次的正确修法 (v0.3.35)

全新重写 `desktop.py`，核心逻辑如下：

```python
# 在 chromadb 被导入之前，同时预注入两层模块
if getattr(sys, "frozen", False):  # 仅在打包环境生效，不影响开发
    sys.modules.setdefault("chromadb.utils", _chroma_utils)
    sys.modules.setdefault("chromadb.utils.embedding_functions", _chroma_ef)
```

关键改进：
1. **两层都注入**：`chromadb.utils` 和 `chromadb.utils.embedding_functions` 同时注入，彻底打断循环链路。
2. **`setdefault` 而非强制覆盖**：正常开发环境下真实的 chromadb 模块优先，不影响本地调试。
3. **父子关系正确建立**：`_chroma_utils.embedding_functions = _chroma_ef`，模块层级正确，不会触发额外的 `AttributeError`。
