流式输出（Streaming）无法实现，通常卡在三个环节中的某一个：**llama.cpp 后端配置**、**中间件 API 转发**，或者 **前端接收逻辑**。
流式输出的本质是服务器不再一次性把答案塞进一个大的 JSON 包，而是通过 **SSE（Server-Sent Events）** 协议，像“挤牙膏”一样把生成的每一个 Token 实时推给前端。
我们要排查以下链路：
### 1. llama.cpp 服务端配置
如果你是直接用 llama.cpp 自带的服务器（./server），请确保：
 * **启动参数：** 并没有特定的启动参数限制流式，但请求时必须带上 "stream": true。
 * **测试命令：** 先用 curl 排除后端问题。
   ```bash
   curl http://localhost:8080/completion \
   -H "Content-Type: application/json" \
   -d '{
     "prompt": "写一个关于猫的故事",
     "stream": true
   }'
   
   ```
   *如果终端里是字一个个跳出来的，说明后端没问题。*
### 2. 中间件（Python/Node.js）的“转发瓶颈”
这是最容易出问题的地方。很多开发者在写后端 API 时，虽然请求了 llama.cpp 的流，但在返回给前端时，却无意中**等待流结束才返回**。
#### 如果你用 Python (FastAPI):
必须使用 StreamingResponse，并且确保你是在**迭代** llama.cpp 的返回结果。
```python
from fastapi.responses import StreamingResponse

@app.post("/generate")
async def generate(request: Request):
    # 调用 llama.cpp 的 API
    def stream_generator():
        # 这里假设你用了 requests 或 httpx 请求 llama.cpp
        with httpx.stream("POST", "http://localhost:8080/completion", json=data) as r:
            for line in r.iter_lines():
                if line:
                    yield f"{line}\n\n" # SSE 格式

    return StreamingResponse(stream_generator(), media_type="text/event-stream")

```
#### 常见坑点：
 * **Gzip 压缩：** 如果你的后端开启了 Gzip 压缩，它可能会先缓存一部分内容再压缩发送，导致“流”变“停顿”。**建议对 AI 接口禁用压缩。**
 * **Buffer 缓存：** Nginx 转发时，如果开启了 proxy_buffering，流会被拦下来。记得在 Nginx 配置里加上 proxy_buffering off;。
### 3. 前端接收逻辑（fetch API）
传统的 axios 默认不支持流式处理。在前端，你需要使用原生的 fetch 结合 ReadableStream。
**前端代码示例：**
```javascript
const response = await fetch('/api/generate', {
    method: 'POST',
    body: JSON.stringify({ prompt: "..." }),
});

const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    
    const chunk = decoder.decode(value, { stream: true });
    // 这里处理 chunk，比如追加到笔记编辑器的 state 中
    console.log("收到的片段:", chunk);
    updateEditor(chunk); 
}

```
### 4. 为什么 Notion 效果更好？
Notion AI 在流式输出时做了大量的 **“UI 预补偿”**：
 1. **打字机音效/动画：** 即使模型吐字不稳，前端也会通过平滑动画让文字显示得更有节奏。
 2. **Markdown 增量解析：** AI 吐出一半的 Markdown 源码（比如 **代码...）时，前端渲染引擎（如 react-markdown）需要能实时解析出未闭合的标签。
