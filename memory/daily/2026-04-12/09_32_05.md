没错，咱们的 Nova 编辑器**不是传统的纯 Markdown 编辑器**，而是基于 Block（块级）的富文本编辑器（底层是 Tiptap/ProseMirror 的那一套节点树模型）。如果直接往里面塞 Markdown 的 ```python，它只会把它当成普通文本展示，根本不会变成真正的代码块！

所以，我给咱们设计了一套**「前端拦截 + 原生 API 调用」的黑科技协议**，具体流程是这样的：

1. **后端不吐 Markdown，吐 Action 标签**：
   当触发代码块指令时，后端引擎返回的不是 Markdown 代码，而是一个自定义的 XML 动作标签，比如：
   `<Action type="insert_code_block" language="python">这里是纯代码</Action>`

2. **前端流式解析器（Stream Parser）暗中拦截**：
   在 `NovaBlockEditor.tsx` 里，我写了一个拦截器。当前端监听到 AI 传过来的是 `<Action>` 标签时，它**不会**把这段文字当作普通字符渲染到屏幕上，而是把它拦截下来偷偷解析。

3. **调用编辑器原生 API 创建 Node（节点）**：
   前端拿到 `type="insert_code_block"` 和代码内容后，直接调用编辑器原生的命令（类似于 Tiptap 的 `editor.chain().focus().insertContent({ type: 'codeBlock', attrs: { language: 'python' }, content: [...] }).run()`）。
   这样一来，编辑器就会在底层数据结构里**直接无中生有地插入一个真正的代码块节点（Node）**，并把纯代码填进去！

**这也正是为什么我之前要去修复“多余的反引号”问题**：
因为小模型有时候会在 `<Action>` 标签里面手贱多加几个 ```python，导致通过原生 API 插入到代码块里的内容，自带了多余的反引号。我加了正则把这些反引号扒干净，保证塞进代码块节点里的就是纯净的代码！

所以总结下来就是：**我们是在用 AI 生成“对编辑器的控制指令”，而不是在生成“文本流”！** 酷不酷？！😎
