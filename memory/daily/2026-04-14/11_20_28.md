- **根因分析**：在 `SlashMenuConfig.tsx` 的 `onUpdate` 回调中，直接通过 `popup[0].setProps(...)` 去更新 tippy 实例。但是在某些极端的生命周期（如由于频繁输入导致快速挂载/卸载）下，`popup` 这个变量可能还没有完成初始化（或者是空数组/未定义），导致读取 `popup[0]` 抛出 `undefined is not an object` 的严重错误，直接搞崩整个 React 渲染树。

#### 2. Drag Handle (拖拽手柄) 漂移问题
- **现象**：当内容超过一屏时，如果滚动编辑器所在的自定义滚动容器（而不是整个 window 滚动），拖拽手柄会停留在原来相对视口的位置，跟着滚动条漂移，而不是吸附在对应的 Block 旁边。
- **根因分析**：拖拽手柄使用的是 `@tiptap/extension-drag-handle-react`，其底层依赖了 `@floating-ui/dom` 来计算手柄悬浮的位置。默认情况下，floating-ui 会监听全局 `window` 的 `scroll` 和 `resize` 事件来重新计算坐标。但在咱们 Nova 中，编辑器是被包裹在一个特定的拥有 `overflow-y: auto` 的自定义滚动容器里的。因此，当这个内部容器滚动时，并未触发 floating-ui 的重算逻辑，导致坐标未更新。
- **修复思路**：不应当使用简单的 CSS 偏移去“掩盖”，而应该在我们的自定义滚动容器的 `onScroll` 事件中，找到对应的拖拽手柄插件实例，强制让它（或者触发 window scroll 事件）重新执行 floating-ui 的 `computePosition`。同时配合 hovered block 的位置记忆，确保计算时的参照物正确。
