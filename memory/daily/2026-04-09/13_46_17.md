1. **致命的“重排 (Layout Thrashing)”**：
   侧边栏从 280px 缩到 64px 时，中间的宽度是连续变化的。如果旁边连着那个几千字、带有各种复杂卡片（KaTeX、代码块、表格）的 **Tiptap 富文本编辑器**，侧边栏每一帧的变化，都会逼着整个编辑器重新计算排版（文字换行、图片挤压等）。你疯狂点击，浏览器就要在一秒内做几十次这种海量计算，CPU 直接被榨干，必然掉帧！
2. **React 的“渲染风暴”**：
   频繁切换侧边栏状态（`isCollapsed`），如果外层组件没有做好 `React.memo` 隔离，不仅侧边栏在重绘，还会连带触发右侧整个页面（甚至编辑器框架）跟着无意义地跑 React 的 Diff 算法。
3. **Framer Motion 的动画积压**：
   如果你在一秒内狂点 5 次，但每次动画要 0.3 秒才播完，动画库的物理引擎队列里就会堆积大量的状态机切换指令，最后互相打架，导致动画变卡甚至短暂死锁。

**我的外科手术方案（马上执行）：**
- **方案 A (阻断连带伤害)**：把编辑器外层容器的宽度变化从“受侧边栏真实宽度挤压”改成“CSS GPU 加速（Transform 偏移 / absolute定位配合过渡）”，或者用纯 CSS 的 `transition` 代替每一帧的 JS 渲染，让富文本引擎在侧边栏收缩时“无感”。
- **方案 B (防抖与动画精简)**：对快速点击做节流（Throttle），动画没播完前不响应新的点击（防止堆积）；同时排查 SidebarTree 里的 `layout` 属性，去掉非必要的 JS 物理动画计算，改用 CSS `transition` 降低负担。
