### 核心升级点解析与创意拓展<!-- 标题序号: 2.1 --><!-- END_BLOCK_14 -->

<!-- BLOCK_15 | doxcn5ozk9YR2ahZDfmXAaXE6yc -->
<grid cols="2">
<column width="50">
  <callout icon="bulb" bgc="5" bc="5">
  #### 1. 多端同步- **核心诉求**：数据在 Windows 客户端、Web 端、移动端之间无缝流转。
  - **脑暴拓展**：
  	- **离线优先 (Offline-First)**：即使没有网络，所有操作依然流畅，待联网后自动同步。
  	- **实时协作 (可选未来)**：允许多个设备同时在线编辑同一篇笔记，光标实时可见。
  	- **同步状态可视化**：在 UI 上清晰展示“正在同步”、“同步完成”、“离线模式”等状态。
  </callout>
</column>
<column width="50">
  <callout icon="bulb" bgc="5" bc="5">
  #### 2. 外网访问与安全- **核心诉求**：在任何地方都能安全地访问自己的“第二大脑”。
  - **脑暴拓展**：
  	- **零信任网络 (Zero Trust)**：默认不信任任何访问，每次请求都需验证。
  	- **设备指纹与管理**：可信设备列表，新设备登录需授权。
  	- **地理位置登录告警**：异地登录时发送通知，增加安全感。
  	- **审计日志**：记录所有访问和重要操作，便于追溯。

  - **脑暴拓展**：
  	- **端到端加密 (E2EE)**：确保只有你自己能解密笔记内容，即使是我们也无法窥探。

### 重点脑暴：游戏化与灵感捕捉 (第 6、7 点)<!-- 标题序号: 2.2 --><!-- END_BLOCK_18 -->

<!-- BLOCK_19 | doxcn1qmx6Vl6euCtSwbJDFY3Ug -->
这是我们本次升级中最具创造性的部分，旨在将你的“二次元/游戏宅”特质深度融入产品体验。
<!-- END_BLOCK_19 -->

<!-- BLOCK_20 | doxcnVQJ9U2T7SkxKMB1dLRpUKf -->
<callout icon="star" bgc="3" bc="3">
#### **核心理念：将“知识管理”转化为一场“角色扮演游戏 (RPG)”**你的每一次记录、学习、思考，都是在为你扮演的“知识探索者”角色积累经验、解锁技能、完成任务。
#### **A. 灵感碎片捕捉：打造无处不在的“拾取”体验****目标**：让记录灵感像在游戏中“捡道具”一样轻松、即时、有快感。
- **全局快捷键 (桌面端)**：
	- `Alt + Q` (QuickThought)：立即弹出一个极简便签窗口，输入后即刻消失，自动存入“灵感暂存箱”。输入时可伴有清脆的“金币+1”音效。
- **桌面悬浮宠物 / 看板娘**：
	- 一个可爱的二次元角色（可选，如 chibi 版的动画角色）常驻桌面。
	- **拖拽投喂**：将网页链接、文字、图片文件拖到它身上，即可快速收藏，宠物会做出开心的动画反馈。
	- **语音速记**：双击宠物，开启语音识别，说完自动转文字存入“灵感暂存箱”。可以说“Aime 酱，帮我记一下...”。
- **游戏风格 UI 浮窗**：
	- 快捷键或看板娘触发的输入窗口，可以设计成类似《女神异闻录5》的对话框风格，动感十足。
- **移动端快速入口**：
	- **系统分享菜单集成**：在任何 App 中，通过系统分享功能，直接将内容发送到 Second Brain。
	- **语音助手集成**：设置 Siri / 小爱同学等语音快捷指令，例如“用第二大脑记个东西”。
#### **B. 游戏化系统：你的知识成长之旅**- **成就系统 (Achievement System)**：
	- **类型**：分为“探索”、“收藏家”、“创造者”、“连接者”等系列。
	- **示例**：
		- `[初窥门径]`：完成第一篇笔记。
		- `[知识炼金术士]`：创建 100 个双向链接。
		- `[深夜的思考者]`：在凌晨 1-3点 连续记录 7 天。
		- `[次元旅者]`：收藏超过 50 个来自 Bilibili 或 Pixiv 的链接。
		- `[稀有掉落]`：解锁一个隐藏的限定版主题。
- **经验值 (EXP) 与等级**：
	- **获取**：新建笔记、建立链接、完成模板、每日回顾等行为都会获得 EXP。
	- **升级**：等级提升会解锁新功能、新的看板娘服装或主题。
- **技能树 (Skill Tree)**：
	- **概念**：将知识管理的方法论（如 Zettelkasten、PARA）转化为可视化的技能树。
	- **点亮技能**：通过实践来“点亮”技能点。例如，当你创建了足够多的“原子笔记”并用索引将它们连接起来后，技能树上的“卡片盒笔记法”技能就会被点亮，并可能为你提供更高级的链接推荐功能。
- **看板娘 (Live2D Companion)**：
	- **互动**：她会根据你的操作给予反馈。当你长时间未记录时，她可能会打哈欠提醒你；当你完成一个大项目时，她会为你鼓掌。
	- **换装**：通过成就或等级解锁新服装，打造你的专属看板娘。
	- **背景故事**：为看板娘设定一个简单的背景故事，她是你的“知识向导”或“记忆精灵”。
</callout>

### 1. 技术路线对比<!-- END_BLOCK_25 -->

<!-- BLOCK_26 | doxcnGSRA49vtt3lxV75quo0kdf -->
- **路线 A (Python + Electron)**：
	- **优势**：可充分利用 Python 强大的生态（如数据分析、AI 集成），性能较好，适合复杂后端逻辑。
	- **劣势**：打包体积大，跨平台一致性维护成本稍高。
<!-- END_BLOCK_26 -->

<!-- BLOCK_27 | doxcn3P99m4IunTUnceX1PXS8Ve -->
- **路线 B (纯前端 IndexedDB)**：
	- **优势**：轻量，天生跨平台（Web/Electron），易于维护和部署。
	- **劣势**：性能受限于浏览器环境，复杂计算（如本地 AI 模型）需依赖 WebAssembly，生态相对不成熟。
<!-- END_BLOCK_27 -->

<!-- BLOCK_28 | doxcn0tHnhTYHEnmcZ6MRomIyrg -->
**结论**：在第一阶段，我们将**并行探索和验证两条路线**，特别是针对多端同步和数据存储的核心模块，构建原型以评估实际性能和开发体验，最终选择更适合长期演进的方案。
<!-- END_BLOCK_28 -->

### 2. 核心技术点分析<!-- END_BLOCK_29 -->

<!-- BLOCK_30 | doxcnGrw4fbZi7oaA5tDNXQ5Lib -->
<table header-row="true" col-widths="150,300,300">
    <tr>
        <td>**功能点**</td>
        <td>**技术实现方案**</td>
        <td>**双路线兼容性说明**</td>
    </tr>
    <tr>
        <td>**多端同步**</td>
        <td>- **核心机制**：采用**离线优先**架构，所有操作立即响应并存入本地。
- **冲突解决**：
	- **方案 1 (推荐)**：**CRDT (无冲突复制数据类型)**。各端可独立写入，数学上保证最终一致性，是离线协作的理想模型。
	- **方案 2**：**事件日志 + 版本向量**。记录每次操作，同步时检查版本冲突，若冲突则由服务端或客户端进行合并。
- **同步触发**：App 启动时、网络恢复时、固定时间间隔轮询、或通过 WebSocket 实时推送。</td>
        <td>- **路线 A**：Python 后端负责同步逻辑，与云端服务 API 交互。
- **路线 B**：纯 JS/TS 实现同步逻辑，直接在客户端内完成。
- **兼容性**：同步逻辑是**应用层协议**，与具体技术栈无关，双路线均可实现。CRDT 库在 Python 和 JS 中都有成熟实现。</td>
    </tr>
    <tr>
        <td>**外网访问与安全**</td>
        <td>- **安全隧道**：采用 **Cloudflare Tunnel** 或 **Tailscale**。它们能在不暴露公网 IP 的情况下，为你的服务创建一个安全的外部访问入口。
- **访问控制**：
	- **身份验证**：所有访问必须通过 OAuth 2.0 / JWT 令牌验证。
	- **设备绑定**：首次登录的设备需要通过邮件或二次验证进行授权，生成设备唯一令牌。
	- **ACL (访问控制列表)**：在隧道层面配置，只允许通过身份验证的用户访问。</td>
        <td>- **路线 A/B**：该层面的安全策略与客户端技术栈无关，主要在云端和网络基础设施层面配置。两种路线的客户端都只需遵循统一的认证和授权流程。</td>
    </tr>
    <tr>
        <td>**私密笔记&nbsp;(E2EE)**</td>
        <td>- **加密算法**：采用 **AES-256-GCM** 对称加密笔记内容。
- **密钥管理**：
	- **主密钥 (MK)**：由用户设置的**口令**通过 **PBKDF2** 或 **Argon2** 派生而来，**绝不上传**。
	- **数据加密密钥 (DEK)**：每篇笔记使用一个随机生成的 DEK。
	- **加密 DEK**：DEK 使用 MK 加密后，与加密笔记一同存储。
- **恢复策略**：提供一个由用户妥善保管的“恢复代码”，该代码可用于在忘记口令时重置并重新加密数据。</td>
        <td>- **路线 A**：加密/解密操作可在 Python 后端执行，利用成熟的密码学库。
- **路线 B**：使用 Web Crypto API 在浏览器/Electron 环境中执行加密/解密，密钥同样仅存于客户端内存。
- **兼容性**：核心密码学逻辑可封装，保证双路线实现一致。</td>
    </tr>
    <tr>
        <td>**时间轴功能**</td>
        <td>- **事件模型**：定义统一的事件结构（`Event{id, type, timestamp, payload, metadata}`）。所有操作（笔记创建/编辑、链接添加、灵感捕捉）都记录为事件。
- **数据索引**：在本地数据库 (SQLite/IndexedDB) 中对 `timestamp` 和 `type` 字段建立索引，加速查询。
- **聚合与过滤**：后端或前端提供 API，支持按时间范围、事件类型、关键词进行高效查询和聚合。</td>
        <td>- **路线 A**：SQLite 提供更强大的索引和查询能力，适合复杂聚合。
- **路线 B**：IndexedDB 同样支持索引，但复杂查询性能可能稍弱，可通过冗余数据或预计算视图优化。
- **兼容性**：数据模型统一，查询逻辑可适配两种数据库。</td>
    </tr>
    <tr>
        <td>**笔记模板**</td>
        <td>- **模板存储**：模板作为一种特殊的笔记类型存储，内容为 Markdown 文本。
- **变量渲染**：使用简单的占位符语法，如 `{{DATE}}`、`{{TIME}}`。在创建新笔记时，客户端进行字符串替换。
- **版本化**：利用现有的笔记版本控制机制，模板的修改也支持回滚。
- **模板仓库**：未来可构建一个独立的 Git 仓库或云端服务来管理和分发模板。</td>
        <td>- **路线 A/B**：模板渲染是纯客户端逻辑，双路线实现方式几乎一致。</td>
    </tr>
    <tr>
        <td>**自定义主题/图标**</td>
        <td>- **主题引擎**：基于 **CSS 自定义属性 (CSS Variables)** 构建。定义一套设计令牌（Design Tokens），如 `--primary-color`, `--font-family`。
- **一键换装**：切换主题时，只需加载一个新的 CSS 文件，覆盖这些变量的定义即可。
- **图标包**：支持 SVG Sprite 或字体图标。切换时，替换对应的资源文件或 CSS 定义。
- **安全**：对用户上传的主题包进行安全扫描，防止注入恶意 CSS 或脚本。</td>
        <td>- **路线 A/B**：这是纯前端的范畴，双路线共享完全相同的实现机制。</td>
    </tr>
</table>

### TDD 驱动的开发步骤<!-- 标题序号: 4.3 --><!-- END_BLOCK_43 -->

<!-- BLOCK_44 | doxcn6fjCohufFzZP7KnLZ2OZOh -->
我们将严格遵循 TDD 流程，确保每一步都有测试保障。
<!-- END_BLOCK_44 -->

<!-- BLOCK_45 | doxcnzLLyjkAeGwTOicx2IktjAc -->
#### **Sprint 1 (第 1-2 周：2026-03-25 ~****&nbsp;2026-04-07)：环境搭建与原型验证**<!-- 标题序号: 4.3.1 --><!-- END_BLOCK_45 -->

<!-- BLOCK_46 | doxcn8BNk8FcmNxEMZRcf8pMwmf -->
1. **任务**：搭建双路线开发环境，完成数据模型定义和本地数据库读写原型。
<!-- END_BLOCK_46 -->

<!-- BLOCK_47 | doxcnzQFpsKP6ibjo3reTFsp2vg -->
2. **TDD 步骤**：
	- **测试类型**：单元测试。
	- **RED**：编写一个测试，尝试创建一个笔记对象并保存到本地数据库（SQLite/IndexedDB），断言无法读出。
	- **GREEN**：实现最简单的数据库初始化和笔记写入逻辑，使测试通过。
	- **REFACTOR**：抽象数据库操作接口 `IDatabase`，使上层逻辑与具体实现解耦。
	- **验收标准**：两条路线均可通过接口完成笔记的增删改查，所有单元测试通过。
<!-- END_BLOCK_47 -->

<!-- BLOCK_48 | doxcnZEOnZQEWmefGz4CcIHWpMg -->
#### **Sprint 2 (第 3-4 周：2026-04-08 ~****&nbsp;2026-04-21)：同步核心与云端服务**<!-- 标题序号: 4.3.2 --><!-- END_BLOCK_48 -->

<!-- BLOCK_49 | doxcnMJhXN7BaR5Vqm5eaDjtU8e -->
1. **任务**：实现核心同步逻辑 (CRDT) 和云端基础服务 API。
<!-- END_BLOCK_49 -->

<!-- BLOCK_50 | doxcngwUe8g4aMKz73pRA3qkYFe -->
2. **TDD 步骤**：
	- **测试类型**：集成测试。
	- **RED**：编写测试模拟两个客户端（Client A, B）。A 离线修改笔记，B 在线修改同一笔记。A 上线后同步，断言 A 的最终结果与 B 不一致（或同步失败）。
	- **GREEN**：实现 CRDT 合并逻辑或事件日志冲突解决机制，确保 A 同步后能与 B 达到最终一致状态，使测试通过。
	- **REFACTOR**：将同步逻辑封装为 `SyncEngine` 模块，清晰化其与网络层、数据层的交互。
	- **验收标准**：在模拟网络分区和延迟的环境下，多端数据同步的冲突解决成功率达到 99.9%，所有集成测试通过。
<!-- END_BLOCK_50 -->

<!-- BLOCK_51 | doxcnE7SVvSM3macFCdNk3z2Dhb -->
#### **Sprint 3 (第 5-6 周：2026-04-22 ~****&nbsp;2026-05-05)：安全隧道与私密笔记**<!-- 标题序号: 4.3.3 --><!-- END_BLOCK_51 -->

<!-- BLOCK_52 | doxcnHq9uXOg38oX26F764wZAQe -->
1. **任务**：配置 Cloudflare Tunnel，并实现端到端加密功能。
<!-- END_BLOCK_52 -->

<!-- BLOCK_53 | doxcniidzURSPWk0gIWFzDlCn2c -->
2. **TDD 步骤**：
	- **测试类型**：安全测试、单元测试。
	- **RED**：编写测试，尝试在未授权的情况下访问服务 API，断言被拒绝。编写另一测试，加密一段文本后，用错误的密码尝试解密，断言失败。
	- **GREEN**：配置好 Tunnel 的访问策略，确保只有携带有效 JWT 的请求才能通过。实现基于用户口令的密钥派生和加解密逻辑，使测试通过。
	- **REFACTOR**：将加解密功能封装为 `CryptoUtil`，并确保没有敏感信息（如明文密码）硬编码或出现在日志中。
	- **验收标准**：通过第三方渗透测试工具扫描，未发现明显漏洞。数据在传输和存储层均为密文。
<!-- END_BLOCK_53 -->

<!-- BLOCK_54 | doxcn5McMf97XFbHt9iA4Reb2ke -->
#### **Sprint 4 (第 7-8 周：2026-05-06 ~****&nbsp;2026-05-19)：Windows 客户端 MVP 组装与回归**<!-- 标题序号: 4.3.4 --><!-- END_BLOCK_54 -->

<!-- BLOCK_55 | doxcnx7u3MON9RCm4i9Gv81KV1c -->
1. **任务**：集成所有模块，构建可交付的 Windows 客户端 MVP，并进行全面回归测试。
<!-- END_BLOCK_55 -->

<!-- BLOCK_56 | doxcnzmkM8e7mn3mSgCCJKcvDnb -->
2. **TDD 步骤**：
	- **测试类型**：端到端 (E2E) 测试。
	- **RED**：编写 E2E 测试脚本，模拟完整用户流程：启动应用 -> 创建私密笔记 -> 断开网络 -> 编辑笔记 -> 恢复网络 -> 检查另一端数据是否同步。断言流程中断或数据不一致。
	- **GREEN**：修复在集成过程中发现的 bug，打通所有环节，使 E2E 测试通过。
	- **REFACTOR**：优化应用启动速度和资源占用。
	- **验收标准**：打包后的 Windows 客户端可在干净环境中成功安装并运行。核心功能（离线编辑、同步、加密）稳定可用，无重大 bug。**所有历史测试用例必须全部通过**，确保零破坏。
<!-- END_BLOCK_56 -->
