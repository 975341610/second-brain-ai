### 二、 技术选型：前端跑还是后端跑？

针对 Nova 项目的不同架构（Web 版、Electron 桌面版或移动端），建议如下：

<grid cols="2">  
<column width="50">  
  **方案 A：前端本地推理 (WebGPU)**  
  - **技术栈**：Transformers.js v3, MLC LLM, WebLLM  
  - **优点**：  
    - **零成本**：无需服务器 GPU，利用用户本地显卡。  
    - **强隐私**：数据不出本地，完全符合笔记应用的私密性需求。  
    - **无缝集成**：纯 JS 驱动，无需安装额外驱动。  
  - **缺点**：首次加载需下载模型（约 1.2GB - 1.5GB）。  
</column>  
<column width="50">  
  **方案 B：后端/桌面侧本地推理 (Local Server)**  
  - **技术栈**：llama.cpp (Wasm/Python), Ollama, Ollama-JS  
  - **优点**：  
    - **性能高**：可利用 C++ 深度优化的推理库。  
    - **稳定性**：不依赖浏览器 WebGPU 的兼容性。  
  - **缺点**：  
    - 增加应用分发体积。  
    - 部署相对复杂。  
</column>  
</grid>

**建议决策：** 如果 Nova 是 Web 或 Electron 应用，**方案 A (WebGPU)** 是当前技术趋势下的最优解，用户体验更接近“原生 AI”。

### 三、 集成挑战与应对策略

<callout icon="thought_balloon" bgc="3">  
集成轻量化模型并非全无门槛，需要重点关注以下三个方面。  
</callout>

1.  **模型体积与下载 (1.2GB - 1.8GB)**
    - **挑战**：用户可能不愿意等待长时间下载。
    - **策略**：
        - **分步加载**：首次仅下载 20MB 的微型 embedding 模型用于搜索，在用户触发总结时异步下载推理模型。
        - **PWA/IndexedDB 缓存**：确保模型只下载一次，永久存储在本地。
        - **4-bit 量化**：强制使用 Q4_K_M 或 Q4_0 量化，将体积压缩到 1.3GB 左右。

2.  **内存与显存占用**
    - **挑战**：2B 模型运行需约 2GB-3GB 内存，低端手机或老旧 PC 可能会卡顿。
    - **策略**：**动态精度调节**。检测设备硬件，低性能设备自动降级至 3-bit 量化或切换到更小的模型（如 Gemma 4 1B，如果发布）。

3.  **首次加载速度**
    - **挑战**：WebGPU 编译 Shader 可能耗时 5-10 秒。
    - **策略**：利用 **Web Workers** 在后台预热模型，并在 UI 上展示友好的“AI 大脑启动中”进度条。

### 四、 行动建议：MVP 方案（第一步）

建议从“笔记自动总结”场景切入，快速验证可行性：

1.  **环境准备**：在 Nova 前端集成 `Transformers.js` v3 (支持 WebGPU)。
2.  **模型获取**：从 Hugging Face 获取 `onnx` 格式的 `Gemma-4-E2B` 量化模型。
3.  **核心实现**：
    - 实现一个 `Summarizer` 类，封装模型下载、加载和推理逻辑。
    - 推理逻辑采用 Web Worker，避免阻塞 UI。
4.  **UI 集成**：在笔记编辑器的侧边栏添加一个“AI 总结”按钮。
5.  **性能监控**：记录不同设备上的 `Time to First Token (TTFT)`，为后续优化提供数据支持。

---
